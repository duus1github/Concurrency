# 第8章 并发程序性能优化
  并法不代表越快，不合理写法反而更慢。本章聚焦多线程性能瓶颈定位+优化手段，
  解决：**锁竞争**，**伪共享**、**上下文切换**、**缓存失效**、**线程数量不合理**等工程问题。
## 核心概念
### 1、性能瓶颈来源
#### 1.1、定义
    多线程程序执行过程中和和会拖慢执行速度，有各类隐性开销
#### 1.2、原理
    CPU缓存架构，内核调度，锁机制，内存可见性都会产生额外耗时
#### 1.3、作用
    识别瓶颈，针对性优化，让并发发挥多核优势

### 2、锁竞争
#### 2.1、定义
多个线程同时争抢一把互斥锁，未抢到则进入阻塞排队
#### 2.2、原理
锁在同一个时刻，只允许一个线程进入临界区，大量线程等待造成串行化执行
#### 2.3、优化方向
**拆分锁**，**缩小锁范围**，**读写锁替换独占锁**

### 3、伪共享
#### 3.1、定义
多个线程独立修改同一CPU缓存行内不同变量，引发缓存频繁失效同步。
#### 3.2、原理
CPU缓存以缓存行为单位同步，一行内任意变量修改，整行缓存失效。
#### 3.3、作用
高频多线程计数，状态标记场景头号性能杀手，必须内存对齐隔离。

### 4、上下文切换
#### 4.1、定义
操作系统暂停当前线程，保存现场，切换执行另一线程
#### 4.2、原理
线程时间片耗尽、阻塞、休眠都会触发切换，消耗内核态资源
#### 4.3、作用
线程数量远超CPU核心数时切换爆炸式增长，性能暴跌。

### 5、缓存局部性
#### 5.1、定义
CPU优先访问高速缓存中靠近、连续的数据。
#### 5.2、原理
顺序访问命中率远高于随机散列访问
#### 5.3、优化
调整数据排布，提升缓存命中率，减少访存延迟

### 6、读写锁
#### 6.1、定义
std::shared_mutex多读单写同步机制
#### 6.2、原理
读共享、写独占，大量读少量写场景大幅降低竞争
#### 6.3、作用
日志读取，配置读取，状态查询场景首选

### 7、任务粒度
#### 7.1、定义
单个任务执行的耗时大小
#### 7.2、原理
任务太小：线程调度开销>业务耗时；任务太大：并发利用率低
#### 7.3、优化
合理拆分/合并任务，均衡负载,
问题:为啥目前案例中是粗锁的时间快于细锁.
原因:
    1.细粒度锁 != 一定更快
    2.细粒度锁 = 更高并行度,但也会带来更多锁开销
    3.这里的业务逻辑过于简单,然后加锁的开销>业务逻辑开销,锁的次数翻倍了,耗时就变高了.
什么时候细锁会更快?
- 当业务逻辑很重,锁竞争很激烈时


## 核心解读
### 1、并发性能黄金原则
- 优先串行优化，再作多线程拆分
- 尽量减少共享数据，从根源减少同步
- 临界区代码越短越好，禁止锁内sleep / IO /复杂计算

### 2、线程数量最优值
CPU密集型：线程数 == CPU物理核心数
IO密集型：可适度多于核心数，利用等待时间跑任务。

### 3、伪共享解决方案
- c++17 std::hardware_destructive_interface_size作内存填充对齐
- 独立变量拆分到不同缓存行

### 4、锁优化层级
- 缩小锁范围
- 拆分全局锁为细粒度分片锁
- 独占锁改读写锁
- 无锁原子操作替换轻量计数

### 5、减少等待开销
- 条件变量合理唤醒，避免频繁惊群
- 自旋等待代替短时阻塞
- 批量提交任务减少队列交互次数
